# OMP (oh-my-pi) agent support — design

Date: 2026-07-02
Status: approved (magenta accent + vendored icon confirmed in session)

## Problem

cmux ≥0.64 recognises OMP ([oh-my-pi](https://github.com/can1357/oh-my-pi),
binary `omp`) as a first-class agent, distinct from Pi: its `coding_agents`
id / hook event source is `omp`. Muxboard's `AgentKind` is a closed union of
`claude | codex | pi | unknown`, so every OMP pane renders as the neutral grey
"?" / AGENT tile and cannot be filtered on dial 2.

## Goal

OMP panes get their own visual identity (real oh-my-pi icon, magenta accent),
are detected from both the authoritative process id and pane titles, and get a
slot in the dial-2 agent filter cycle.

Non-goals: no cmux-side changes (hook install is a user action, documented in
README troubleshooting only if needed); no CodexBar usage integration (OMP has
no CodexBar provider).

## Design

### 1. Type + enumeration sites

- `src/core/types.ts` — `AgentKind` gains `"omp"`.
- `src/config.ts` — `ALL_AGENTS` gains `"omp"` (alias-map validation).
- `src/core/services/store.ts` — `FILTER_CYCLE` becomes
  `["all", "claude", "codex", "pi", "omp"]`.

### 2. Detection

- `src/core/cmux/agents.ts` `toAgentKind()` — `"omp"` → `"omp"`
  (authoritative path: cmux `top --processes` `coding_agents[].id`).
- `src/core/orca/normalize.ts` `toAgentKind()` — same mapping for the Orca
  attention source.
- `src/core/cmux/normalize.ts` `detectAgent()` — title fallback. Matches
  exact `"omp"`, word-boundary `omp` (same boundary class as the pi rule),
  or the substring `oh-my-pi`/`oh my pi`. **Ordering constraint:** the omp
  checks run *before* the pi rule, because the existing pi word-boundary
  regex matches the trailing `-pi` in `oh-my-pi` and would misclassify it.

### 3. Visual identity

- `src/core/render/palette.ts` — new theme:
  `omp: { bg: ["#33142a", "#1f0a19"], accent: "#e070bd", fg: "#f7e6f1", glyph: "Ω", label: "OMP" }`.
  Magenta was chosen over oh-my-pi's brand orange (`#f97316`) for
  glanceability: claude already owns the orange family on the board. Unique
  hue per agent: orange / teal / purple / magenta.
- Vendored icon: oh-my-pi's `assets/icon.svg` (MIT) — a π mark built from
  rects with a plug connector. Flattened to the key-tile treatment used by
  the claude/codex logos: shapes `fill="white"`, plug slots dark, decorative
  dots dropped. `keyRender` already prefers `PROVIDER_ICONS[agent]` over the
  text glyph, so adding an `omp` entry lights it up with no render changes.
  The `Ω` glyph remains as the fallback path.

### 4. Icon pipeline

`providerIcons.ts` is AUTO-GENERATED by `scripts/gen-provider-icons.mjs` from
CodexBar's bundled SVGs. To keep it fully generated:

- Add `assets/provider-icons/omp.svg` — the flattened vendored copy, with a
  source/license comment.
- Extend `gen-provider-icons.mjs` to merge every SVG in
  `assets/provider-icons/` (repo-local overrides) after the CodexBar sweep,
  then regenerate. Local files always exist, so regeneration on a machine
  without CodexBar still emits the omp entry.

### 5. Tests

- `detectAgent`: `"omp"`, `"omp-agent"`, `"oh-my-pi"` → `"omp"`;
  `"stomp"`, `"compare"` stay `"unknown"`; `"oh-my-pi"` must NOT be `"pi"`.
- cmux + orca `toAgentKind("omp")` → `"omp"`.
- Filter cycle includes `"omp"` between `"pi"` and wrap-around to `"all"`.
- `agentTheme("omp")` returns the magenta theme.
- Key render for an omp item embeds the vendored icon body (not the `Ω`
  text fallback).
- `scripts/validate.ts` fixture check extended if a fixture gains an omp
  notification (optional; keep existing checks green).

### 6. Docs

README: add OMP to the supported-agent list and a one-line note that
`cmux hooks omp install` is required for cmux to see OMP sessions at all.

## Error handling

No new failure modes: detection falls through to `"unknown"` exactly as
today; a missing icon entry falls back to the text glyph; the alias map can
still remap arbitrary title substrings to `"omp"`.
